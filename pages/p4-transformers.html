Transformers & LLMs

The heaviest interview block at both product MNCs and AI startups. Amazon's 2025 Bar Raiser probed: attention O(N²) complexity, Flash/Sparse Attention, BERT vs GPT, masked vs causal LM, and tracing a single token through transformer layers — all in one 75-minute round.

Q
Attention Mechanism (Q/K/V math)

Intuition First

Attention lets each position in a sequence "look at" all other positions to decide what information it needs. Q = "what am I looking for?" K = "what information do I have?" V = "what I'll share if selected." The score QKᵀ measures relevance; softmax turns scores into weights; the output is a weighted blend of V vectors.

Scaled dot-product attention formula:

Attention(Q, K, V) = softmax(QKᵀ / √d_k) · V

Q ∈ ℝ^(n×d_k), K ∈ ℝ^(n×d_k), V ∈ ℝ^(n×d_v). QKᵀ ∈ ℝ^(n×n) — the attention weight matrix.

Why scale by √d_k? With high d_k, dot products grow large in magnitude (variance = d_k). Large values → softmax outputs near 0 or 1 → sparse distribution → near-zero gradients for most keys → slow/no learning. Dividing by √d_k keeps variance ~1 → softmax stays in useful gradient range. This is why the scale factor is critical, not cosmetic.

O(N²) complexity: Computing QKᵀ for sequence length N: N×N matrix. Both time and memory are O(N²·d_k). For N=1000: 1M attention scores. For N=100k (long documents): 10B scores — infeasible. This is why context length is a limitation and why Flash/Sparse attention exist.

Causal (masked) attention: In GPT-style decoders, each token can only attend to past tokens, not future. Achieved by masking future positions: set upper triangle of QKᵀ to -∞ before softmax → softmax output → 0 for future positions. This enables autoregressive generation: generate token by token, each attending to all previous.

What do Q, K, V actually represent? All three are linear projections of the same input (self-attention): Q = X·W_Q, K = X·W_K, V = X·W_V. The projection matrices W_Q, W_K, W_V are learned. Different heads learn different projections → can attend to different types of relationships simultaneously.

The Interrogation

Killer What happens if you remove the √d_k scaling factor? ▼

Without scaling, dot products QKᵀ have variance = d_k. For d_k=64: std ≈ 8. Softmax of values like [8, -3, 6, -5] is nearly a one-hot vector: one attention weight ≈ 1, rest ≈ 0. This means: (1) Gradient vanishing: softmax derivative ≈ 0 for non-attended positions → most attention weights receive near-zero gradients → slow learning. (2) Loss of soft attention: Instead of a weighted blend of V, you get a nearly hard selection. This is information-poor. (3) Training instability: Very sharp attention distributions early in training can create feedback loops. The scaling keeps Var(Q·K) ≈ 1 regardless of d_k, keeping the softmax in its informative gradient region.

Depth Probe Trace a single token through a complete transformer layer (Amazon asked this). ▼

Input: token embedding x ∈ ℝ^d (d=768 for BERT-base).

1. x̃ = LayerNorm(x) [pre-norm variant] 2. Q=x̃·W_Q, K=x̃·W_K, V=x̃·W_V [project to Q/K/V, each ℝ^(d_k)] 3. scores = QKᵀ / √d_k [n×n matrix for sequence of n tokens] 4. scores = mask(scores) [causal: -inf for future positions] 5. weights = softmax(scores) [n×n attention weights] 6. attn_out = weights · V [weighted sum of values] 7. attn_out = concat(heads) · W_O [multi-head: concat all heads, project] 8. x = x + attn_out [residual connection] 9. x̃ = LayerNorm(x) [pre-norm for FFN] 10. ffn_out = GELU(x̃·W_1 + b_1)·W_2 + b_2 [FFN: d→4d→d, 2 linear layers] 11. x = x + ffn_out [residual connection] Output: x ∈ ℝ^d for this token position

Repeat across L layers. Final output fed to task head (classification) or language model head (softmax over vocab).

H
Multi-Head Attention & Why It Works

Multi-head attention formula:

MultiHead(Q,K,V) = Concat(head_1,...,head_h) · W_O where head_i = Attention(Q·W_Q_i, K·W_K_i, V·W_V_i)

Each head uses its own W_Q_i ∈ ℝ^(d×d_k), W_K_i ∈ ℝ^(d×d_k), W_V_i ∈ ℝ^(d×d_v) where d_k = d_v = d/h. So 8 heads with d=512 → d_k=64 per head. Concatenated output is d → projected back to d via W_O.

Why multiple heads instead of one big head? A single attention head computes a single weighted blend of values — it can focus on one type of relationship at a time. Multiple heads can simultaneously attend to: syntactic dependencies (subject-verb), semantic relationships (coreference), positional patterns, rare/specialized patterns. Each head learns a different subspace projection. Information is richer because different aspects of the input contribute independently.

Computational cost: Multi-head is NOT more expensive than single-head with the same total dimension. Single head with d_model dimensions: O(N²·d_model). Multi-head with h heads of d_k=d_model/h each: h × O(N²·d_k) = O(N²·d_model). Same compute. The heads are parallelized across the GPU.

What heads actually learn (empirically): Positional heads, rare-token heads, syntactic dependency heads. Some heads are redundant and can be pruned without loss. In large models, different layers develop different specializations (lower layers: syntax, higher layers: semantics).

The Interrogation

Mid If multi-head attention has the same compute as single-head, why not just use one big head? ▼

Same compute, but very different expressiveness. One head with d_model dimensions projects Q, K, V into a single d_model-dimensional space and learns one attention pattern. Multiple heads each project into a lower-dimensional subspace (d_k = d_model/h) and learn different patterns simultaneously. The subspace constraint is a feature, not a bug: it forces each head to specialize. Concatenating heads allows the model to attend to many different types of information in parallel — positional, semantic, syntactic — and combine them via W_O. A single large head would try to learn all these patterns in one attention distribution, but a single softmax distribution can only focus on one dominant pattern per query position. The mathematical reason: rank of single-head attention = rank of QKᵀ/√d_k is limited. Multi-head, after concatenation, spans a richer subspace. Ablation studies confirm: removing heads hurts performance less than removing the same fraction of parameters elsewhere.

Positional Encodings (sin/cos, RoPE)

Classic Depth Probe

"Why do we need positional encodings at all? What would happen without them?" — standard transformer depth probe. Know the answer from first principles.

Why Transformers need position info: Attention is permutation-equivariant — if you shuffle the input tokens, you get the same output (just shuffled). This means a Transformer has no inherent notion of "first word" vs "last word" without additional position information. NLP is order-dependent ("dog bites man" ≠ "man bites dog"), so positions must be injected.

Sinusoidal positional encodings (original Transformer):

PE(pos, 2i) = sin(pos / 10000^(2i/d_model)) PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

Different frequencies for different dimensions. Low dimensions: high frequency (alternates every few positions). High dimensions: low frequency (changes slowly over long sequences). Properties: unique encoding per position, predictable relative distances, can generalize to longer sequences than seen during training (extrapolation is imperfect). PE is added to token embeddings, not concatenated.

Learned absolute positional embeddings: Just a lookup table: PE(pos) = embedding table[pos]. Used in BERT, original GPT. Simpler but can't extrapolate beyond max training length.

RoPE (Rotary Position Embedding): Used in LLaMA, GPT-NeoX, Falcon. Instead of adding position info to embeddings, rotate the Q and K vectors by a rotation matrix that depends on position. The key property: dot product Q_pos1 · K_pos2 depends only on relative position (pos1 - pos2), not absolute positions. This is better for generalization to longer sequences. Combined with YaRN or dynamic NTK scaling, RoPE models can extend to longer contexts at inference than trained on.

ALiBi (Attention with Linear Biases): Instead of encoding positions in embeddings, add a bias to attention scores: score -= m·(key_pos - query_pos). m is a head-specific slope. Simpler, naturally handles longer sequences at inference.

The Interrogation

Depth Probe What breaks if you remove positional encodings entirely? Give a concrete example. ▼

Without PE: the Transformer is a "bag of words" model that's order-invariant. "The cat sat on the mat" and "The mat sat on the cat" would produce identical representations for each token (different arrangement, same set). Concrete failure: (1) Language modeling: predicting "the" after "cat sat on the" requires knowing "the" was the most recent context, not just that these words appear together. Without position: can't distinguish which tokens are recent. (2) Coreference: "John said he was tired" — "he" refers to "John" because it comes after. Without position ordering, "he" could refer to anything. (3) Syntax: verb agreement requires knowing which noun is subject (determined by position). Experiments confirm: Transformers trained without PE show degraded performance on order-sensitive tasks but surprisingly retain much capability for bag-of-words tasks, showing that attention alone captures some positional information implicitly through content — but far less effectively.

BERT vs GPT Architecture

Amazon Asked This (2025)

"Explain BERT vs GPT architectural and use-case differences. Masked vs causal language modeling. Trace a single token through a GPT layer." All from Amazon's 2025 ML Deep Dive Bar Raiser round.

BERT (Bidirectional Encoder Representations from Transformers): Encoder-only. Bidirectional — each token attends to ALL other tokens (left and right). Pre-training: Masked Language Modeling (MLM) — randomly mask 15% of tokens, predict them from context. Also: Next Sentence Prediction (NSP). Not autoregressive — can't generate text token by token. Best for: classification, NER, Q&A (reading comprehension), semantic similarity — tasks requiring full context understanding.

BERT: attention mask = all-ones (full bidirectional) MLM: predict masked tokens using full left+right context

GPT (Generative Pre-trained Transformer): Decoder-only. Causal/autoregressive — each token attends only to previous tokens (upper triangle of attention matrix is masked). Pre-training: Causal Language Modeling (CLM) — predict next token given all previous. Can generate text by sampling one token at a time and feeding it back as input. Best for: text generation, summarization, code generation, chat, in-context learning.

GPT: attention mask = lower triangular (causal) CLM: predict x_t given x_1,...,x_{t-1}

Why causal masking enables generation: If a model attends to future tokens during training, at inference it can't — future tokens don't exist yet. Causal training means the model only sees what's available at inference → autoregressive generation is consistent with training.

T5, BART (Encoder-Decoder): Encoder processes input bidirectionally; decoder generates output autoregressively, attending to encoder output via cross-attention. Best for: seq2seq — translation, summarization with explicit input-output separation.

Modern trend: Decoder-only models (GPT style) dominate. Few-shot in-context learning is a GPT strength — the causal attention allows earlier tokens to "set up" the task for later tokens. BERT-style models are still preferred for discriminative tasks.

The Interrogation

Mid Why can't BERT generate text? ▼

BERT's attention is bidirectional — each token attends to all others including future tokens. During pre-training (MLM), the model learns to predict masked tokens using context from both sides. To generate text token by token, you need to predict token t given only tokens 1..t-1 (you don't have t+1..n yet). BERT was never trained on this causal objective — it expects to see all context. If you try to generate autoregressively with BERT, the model will see placeholder/empty future positions, which is a distribution mismatch from training (where there's always a rich full context). It can't generate coherently. GPT was trained with the causal objective from the start — each position predicts the next given only the left context → generation is natural and consistent with training.

KV Cache & Inference Optimization

Amazon Asked This (2025)

"Attention complexity O(N²) and optimization strategies — Sparse Attention and Flash Attention mechanisms." This appeared directly in Amazon's 2025 ML Challenge Bar Raiser round.

The problem without KV cache: Generating token t+1 requires computing attention over all t previous tokens. Without caching, for each new token you'd re-compute K and V for ALL previous positions (re-running the encoder for the entire prefix). For token t in a sequence of length T, that's O(T·d) compute per token, O(T²) total → extremely slow for long sequences.

KV Cache solution: At each generation step, cache the K and V vectors for all previous positions. When generating the new token, only compute Q for the new token, then attend over cached K and V. Cost per new token: O(T·d_k) instead of O(T²). Memory: O(T·n_layers·n_heads·d_k) to store the cache.

Without KV cache: O(T²) total attention compute With KV cache: O(T) per token, O(T²) total but amortized across time

KV cache memory growth: For LLaMA-7B, 4k context: KV cache ≈ 4GB. For 100k context: KV cache ≈ 100GB — larger than the model itself. This is why long context is expensive. Solutions: (1) Quantized KV cache (4-bit → 4× reduction). (2) Multi-Query Attention (MQA): multiple Q heads share a single K/V head — reduces KV cache size by h×. (3) Grouped Query Attention (GQA): G groups of Q heads share K/V — balance between MQA and full MHA. LLaMA-2 uses GQA. (4) Sliding window attention: only cache last W tokens (used in Mistral).

Prefill vs Decode: Prefill phase — process the entire prompt in parallel (full attention, O(N²), but parallelized → fast). Decode phase — generate tokens one at a time (O(N) per token with KV cache). Prefill is compute-bound; decode is memory-bandwidth-bound (reading KV cache from HBM).

The Interrogation

Depth Probe Why does KV cache memory grow with sequence length, and what's the implication for production serving? ▼

KV cache stores, for each layer and each token, the key and value vectors: 2 × n_layers × n_heads × d_k floats per token. For LLaMA-2 7B (32 layers, 32 heads, d_k=128, float16): 2×32×32×128×2 bytes = 524,288 bytes ≈ 0.5MB per token. At 4k tokens: 2GB. At 32k tokens: 16GB. Implications for production: (1) Batch size constrained by KV cache: With 40GB GPU, you can't serve many users with long contexts simultaneously — each user's KV cache occupies GPU memory. (2) PagedAttention (vLLM): Manages KV cache like virtual memory pages — pages in/out to CPU, enables larger effective batch sizes. (3) Latency depends on cache size: Reading large KV cache is memory-bandwidth-bound — throughput limited by HBM bandwidth, not compute. This is why reducing KV cache (MQA, GQA, quantization) matters so much for inference cost.

Flash Attention & Sparse Attention

The memory problem with standard attention: QKᵀ is an N×N matrix. For N=4096: 4096² × 2 bytes ≈ 32MB per layer per head. With many layers and heads: GB of memory just for attention matrices. Plus: writing/reading this matrix to/from HBM (GPU memory) is slow — the bottleneck is I/O, not compute.

Flash Attention (Dao et al., 2022): IO-aware algorithm that computes attention without materializing the full N×N matrix. Key idea: tile the Q, K, V matrices into blocks that fit in fast SRAM (GPU on-chip memory). Process blocks sequentially, using online softmax normalization (incrementally updating the normalization factor). Never write the N×N attention matrix to HBM.

Flash Attention: same math, different memory access pattern - O(N) memory instead of O(N²) - 2-4× faster wall-clock time (due to fewer HBM reads/writes) - SAME output as standard attention (exact, not approximate)

What Flash Attention does NOT do: Does not reduce FLOPs (still O(N²) multiply-adds). Does not approximate — exact same output. The speedup comes entirely from avoiding expensive HBM round-trips. Flash Attention 2 adds parallelism across sequence length; Flash Attention 3 adds FP8 support.

Sparse Attention: Instead of attending to all N positions, each query attends to only a subset (e.g., local window + strided global tokens). Reduces complexity to O(N·√N) or O(N·log N). Examples: Longformer (sliding window + global tokens), BigBird (random + sliding window + global), Mistral (sliding window attention). Does approximate the full attention — trades some quality for longer context.

Flash vs Sparse: Flash Attention = exact attention, faster I/O. Sparse Attention = approximate attention, fewer FLOPs. For most production LLMs: Flash Attention is used (no quality degradation). Sparse Attention used when context length is very long (100k+) and quality tradeoff is acceptable.

The Interrogation

Depth Probe Flash Attention doesn't reduce FLOPs. So why is it faster? ▼

Modern GPUs are compute-bound at small N and memory-bandwidth-bound at large N. Standard attention: compute QKᵀ (O(N²) FLOPs) → write N×N matrix to HBM (O(N²) memory) → read it back for softmax → write again → read for V multiply → write output. Each HBM round-trip for a 4096×4096 matrix takes ~1ms on A100 (HBM bandwidth ~2TB/s). With 32 layers × 32 heads: many GB of HBM traffic just for attention matrices. Flash Attention: tiles Q, K, V into SRAM (shared memory, ~192KB on A100, access ~19TB/s effective). Computes full tile × tile attention in SRAM, uses online softmax to avoid multiple passes. Never writes the N×N matrix to HBM. I/O complexity: O(N²/M) where M is SRAM size. At large N: ~2-4× fewer HBM bytes transferred → 2-4× wall-clock speedup despite identical FLOP count. The bottleneck shifts from memory bandwidth to compute, which is the intended state for a well-optimized GPU kernel.

T
Tokenization & Vocabulary

Why not character-level? Sequences become very long → O(N²) attention cost explodes. Also: harder to learn word-level semantics from character sequences — requires longer-range dependencies.

Why not word-level? Vocabulary explosion — English has ~500k+ words. Each word is a parameter row in the embedding table and softmax head → huge model. Out-of-vocabulary problem: novel words, names, typos, code → UNK token → information loss.

BPE (Byte Pair Encoding): Start with character vocabulary. Iteratively merge most frequent adjacent pair → new token. Repeat until desired vocab size. "lower" → ["l","o","w","e","r"] → merge common pairs → ["low","er"]. GPT-2, GPT-3, LLaMA use BPE. Handles rare words by decomposing into subword units.

WordPiece: Similar to BPE but merges maximize log likelihood of training data (not raw frequency). Used by BERT. Marks non-initial subwords with ##: "playing" → ["play","##ing"].

Vocabulary size trade-offs: Larger vocab (100k+): each token encodes more text → shorter sequences → cheaper attention. But: larger embedding table, larger output softmax head, rarer tokens have less training data → worse embeddings. Smaller vocab (10k): longer sequences, cheaper per-token but more tokens per document. GPT-2: 50k vocab. LLaMA: 32k vocab. GPT-4: ~100k estimated. Modern trend: larger vocabs to handle multilingual text efficiently.

Scaling Laws & Emergent Abilities

Kaplan et al. (2020) scaling laws: Language model loss scales as a power law with model size N, dataset size D, and compute budget C. Loss ∝ N^(-α), D^(-β), C^(-γ). This means: given a fixed compute budget, you can predict the optimal model size and dataset size.

Chinchilla (Hoffmann et al., 2022): Key finding — GPT-3 (175B params, ~300B tokens) was significantly undertrained. Optimal compute allocation: for compute budget C, optimal model size N* ∝ √C and optimal tokens D* ∝ √C. The ratio N*/D* ≈ 1 (tokens ≈ params). Chinchilla 70B trained on 1.4T tokens outperforms GPT-3 175B trained on 300B tokens. Implication: many large models are params-rich but data-poor — training longer on more tokens is more efficient than making the model bigger.

Emergent abilities: Capabilities that are absent at small scale but appear sharply at large scale — not a smooth improvement. Examples: multi-step arithmetic, chain-of-thought reasoning, few-shot in-context learning. Debate: are these truly emergent or just the result of metric choices? Some argue better metrics show smooth improvement. Regardless: larger models qualitatively unlock capabilities (following complex instructions, code generation) that smaller models fail at even with fine-tuning.

The Interrogation

Mid What was Chinchilla's main finding and how did it change how we train LLMs? ▼

Chinchilla (DeepMind, 2022) showed that for a given compute budget, you should spend roughly equal compute on parameters and tokens — each should scale proportionally with compute budget (both ∝ √C). GPT-3 (175B params, 300B tokens) used 175B/300B ≈ 0.58 tokens per param — undertrained. Chinchilla 70B was trained on 1.4T tokens (20× more tokens, ½ the params) and outperformed GPT-3 on most benchmarks at 40% the inference cost. Impact: (1) Smaller models trained longer now preferred over huge models trained briefly. (2) LLaMA 1 (Meta, 2023) took this seriously: 7B model trained on 1T tokens outperforms GPT-3 on many tasks. (3) Data becomes as important as parameters. (4) Inference cost matters — a smaller, well-trained model is cheaper to run than a larger undertrained one at same benchmark performance.
